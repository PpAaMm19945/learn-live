### 2. SVG Overlay Specification

Each SVG overlay file must:
- Match the dimensions of the corresponding PNG base map (check the PNG file at `docs/curriculum/history/Maps/Maps/Map {NNN}.png`)
- Use a `viewBox` that matches the PNG pixel dimensions
- Contain ONLY overlay elements (regions, routes, markers, labels) — the PNG is the base layer
- Use semantic `id` attributes on all interactive elements (e.g., `id="region-mizraim"`, `id="route-cush-migration"`)
- Use semi-transparent fills for highlighted regions (opacity 0.25–0.35)
- Use the project's color system where possible:
  - Mizraim/Egypt: `hsl(0, 70%, 50%)` (red family)
  - Cush/Nubia: `hsl(270, 60%, 50%)` (purple family)
  - Phut/Libya: `hsl(30, 80%, 50%)` (orange family)
  - Canaan/Levant: `hsl(0, 0%, 50%)` (gray family)
  - Trade routes: `hsl(45, 80%, 50%)` (gold/amber)
  - Water features: `hsl(210, 60%, 50%)` (blue family)
- Use `<path>` elements for regions (not `<rect>` or `<circle>`) to match geographic boundaries
- Use dashed `<path>` elements for migration/trade routes with `stroke-dasharray`
- Use `<text>` elements for labels with `font-family="Inter, sans-serif"`
- Group related elements in `<g>` tags with descriptive IDs (e.g., `<g id="hamitic-regions">`, `<g id="migration-routes">`)

### 3. Source Data

Each map's markdown spec is at `docs/curriculum/history/Maps/map_{NNN}_{slug}.md`. The JSON block inside each markdown file contains:
- `geographic_bounds` — the lat/lon extent of the map
- `highlighted_regions` — regions to draw as polygons
- `settlements` — cities/sites to mark with dots + labels
- `migration_routes` / `paths` — directional arrows to draw as paths
- `annotations` — text boxes, legends, cartouches

The corresponding PNG base map is at `docs/curriculum/history/Maps/Maps/Map {NNN}.png` (note: space before number, 3-digit zero-padded).

### 4. Output Location

SVG files go to: `docs/curriculum/history/Maps/overlays/map_{NNN}_overlay.svg`

Create the `overlays/` directory if it doesn't exist.

### 5. Alignment Data

For each SVG, also output an alignment JSON entry to `docs/curriculum/history/Maps/overlays/alignment-manifest.json`. This is an array of objects:

```json
{
  "mapId": "map_001",
  "pngPath": "docs/curriculum/history/Maps/Maps/Map 001.png",
  "svgPath": "docs/curriculum/history/Maps/overlays/map_001_overlay.svg",
  "transform": {
    "translateX": 0,
    "translateY": 0,
    "scaleX": 1,
    "scaleY": 1,
    "rotate": 0
  },
  "status": "draft",
  "notes": "Initial generation. Needs manual alignment with alignment tool."
}
```

Set all transforms to identity (0/0/1/1/0) — the visual alignment tool (Phase 3.3) will be used later for manual adjustment.

### 6. Quality Rules

- Each SVG must be valid XML (parseable by browser SVG renderer)
- Keep file sizes reasonable — prefer `<path>` with simplified coordinates over highly detailed coastlines
- Region polygons should be approximate but recognizable — they will be refined during manual alignment
- All text must be legible at the expected render size (the canvas component renders at ~800–1200px width)
- Include a `<!-- Generated by Jules, Phase 13, Batch X, YYYY-MM-DD -->` comment at the top of each SVG
