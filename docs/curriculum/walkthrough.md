# Walkthrough: Implementing the Curriculum Spine & Parent Interface

This document summarizes the significant leaps we've made in transitioning the "Learn Live" platform from its MVP state to a fully structured, curriculum-driven architecture capable of supporting Pilot Testing.

## Path 1: Curriculum Architecture & Content Refinements
Based on rigorous feedback, we refined the core `math_curriculum_spine.md` and retrofitted all the existing content templates.

### Architectural Additions
- **The Worksheet Structure:** Defined strict rules for how a single "Worksheet" operates, breaking it down into *Worked Example*, *Execution*, *Endurance* (noise insertion), and a cross-strand *Milestone*.
- **Option D (Split Marking):** We formally adopted the "Split Marking" model. The AI handles the objective evaluation of numerical execution tasks, while the Parent retains the moral authority to judge endurance and milestone transfer tasks.
- **Activity Minimum Rules:** Enforced the strict definition that "Encounter" level capacities *must* involve physical activity, and "Own" level capacities *must* involve active production/teaching by the child.
- **Cross-Strand Routing:** Implemented hard gates ensuring a child cannot access a capacity if its prerequisite dependencies are not firmly at the *Execute* level or higher.

### Template Retrofits (Band 2 "Gold Standard")
The single-instance constraint templates for Band 2 (ages 6-9) across all 5 mathematical strands were meticulously updated to include:
- **Formal Milestone Tasks:** Replaced abstract testing with real-world scenarios where the child must recognize the necessary math without explicit labeling. 
- **Retroactive Reasoning Checks:** Ensured every single template, regardless of cognitive level, requires the parent to ask a specific "Why?" question to test actual understanding against memorization.

*Jules (the autonomous agent) is currently using these "Gold Standard" templates to generate the remaining 2,000+ template variations across all developmental bands into JSON files.*

## Path 2: Data Model & Application Wiring
With the theoretical framework locked, we built the actual data pipeline and user interface to support it.

### 1. D1 Database Schema Expansion
We deployed the following schema updates to the production Cloudflare D1 database:
- `Developmental_Bands` & `Strands`
- `Capacities` & `Capacity_Dependencies` (The DAG engine)
- `Constraint_Templates`
- `Learner_Repetition_State` (Tracking Exposure → Execution → Endurance → Milestone)

### 2. JSON Seeding Pipeline
Created a Node build script (`scripts/seed_curriculum.ts`) that will automatically parse the JSON template variations generated by Jules and inject them directly into the D1 schema structure.

### 3. Parent Task View UI (`ParentTaskCard.tsx`)
We completely revamped the `Dashboard.tsx` to handle the new curriculum framework. Parents now see a clean, distraction-free "Task Card" that presents the Constraint Prompt exactly as the spine defines it.
- **Separated Sections:** Highlights what the parent should *do*, what they should *look for* (Success/Failure conditions), and the specific *Reasoning Check* to ask.
- **Contextual Badges:** Shows the exact strand, capacity, cognitive level, and the child's current arc stage.

### 4. Async AI Evidence Capture (`AsyncEvidenceModal.tsx`)
Built the premium Async Capture flow. Parents can now:
1. Tap a button to open their device camera and snap a photo of the child's physical workspace.
2. Tap a recording button to capture a 10-second audio clip of the child explaining their reasoning.
3. Submit the batched evidence to the AI for an asynchronous evaluation report, lowering bandwidth requirements and live-session performance anxiety.

## Path 3: Field Testing Documentation
To ensure the system works with real humans, we prepared the comprehensive framework for a 5-10 family Pilot Test (Band 2). 

The `pilot_testing_plan.md` artifact defines:
- **The Onboarding Guide:** Clear instructions for parents on their role as the "Evidence Witness" and the strict rules against doing the work for the child.
- **Calibration Metrics:** A data-tracking plan to measure whether the *Execution count* is too high (causing fatigue) and whether the *Endurance noise* is too disruptive.
- **Parent Competence Survey:** A structured survey to evaluate if the constraint prompts successfully reduce math anxiety and if the "Reasoning Checks" yield clear evaluation answers.
